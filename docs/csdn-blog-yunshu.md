# 我做了一个 Go + React 的 Kubernetes 运维与告警平台（含 50+ 页面实战截图）

> 项目名：Yunshu  
> 技术栈：Go + Gin + GORM + Casbin + Redis + MySQL + React + TypeScript + Ant Design  
> 适合读者：想做“后台管理 + 项目化权限 + K8s 资源管理 + 告警 + 日志平台”一体化系统的同学

---

## 一、为什么要做这个项目？

在不少团队里，系统管理、权限管理、K8s 运维、告警平台、日志平台往往是分散的：

- 用户权限和业务项目是两套体系，人员变动后容易出权限错配
- K8s 资源管理依赖命令行，排障效率受个人经验影响大
- 告警规则、值班和通知渠道割裂，告警链路难追踪
- 日志采集和检索入口不统一，新同学 onboarding 成本高

所以我做了 `Yunshu`：把“系统管理 + 项目管理 + K8s + 告警 + 日志”放在同一个平台里，目标是**降低运维复杂度，提高排障效率**。

---

## 二、项目定位与核心能力

`Yunshu` 当前覆盖五大能力域：

1. 系统管理：用户、角色、菜单、组织架构、字典、审计日志
2. 权限治理：Casbin 授权 + K8s 三元策略（cluster + namespace + action）
3. 项目管理：项目、成员、服务器、服务、日志源、Agent
4. 告警平台：数据源、规则、值班、静默、策略、通知记录
5. Kubernetes 管理：工作负载、网络、存储、RBAC、CRD/CR、事件

---

## 三、技术架构概览

后端：

- Go（Gin）
- GORM（MySQL）
- Casbin（权限）
- Redis（缓存/状态）

前端：

- React + TypeScript
- Ant Design
- Vite

运行方式：

- 后端命令入口：`server / migrate / seed / logagent`
- 前端独立 dev server
- 通过项目维度组织资源、告警与日志能力

---

## 四、快速启动（本地开发）

```bash
git clone <your-repo-url>
cd yunshu

go mod download
cd web && npm install && cd ..

go run . migrate
go run . seed
go run . server
```

新终端启动前端：

```bash
cd web
npm run dev
```

## 六、数据库设计与排障思路

数据库名：`permission_system`。

项目里已经补充了精细 ER 文档（总览图 + 5 张子图 + SQL 查询入口）：

- `docs/handbook/database/er-diagrams.md`

建议排障路径：

1. 先按“项目编码”确认项目主链路（成员/服务器/数据源）
2. 告警问题优先查“数据源 -> 规则 -> 值班 -> 事件”
3. 权限问题查“用户 -> 角色 -> 授权规则 + 操作日志”
4. K8s 问题查“集群状态 + 三元策略 + 具体资源操作日志”

---

## 七、我在这个项目里踩过的几个坑

- **前端编辑入口分散**：详情页和编辑页混用，容易造成误改；后面统一成“详情只读 + 表单编辑”
- **项目角色语义和全局权限语义冲突**：最终把权限核心收敛到 RBAC/Casbin 与 K8s 三元策略
- **告警链路可观测性不足**：补齐规则、值班、通知记录的可追溯查询入口
- **文档与截图脱节**：README 和 handbook 分层，主文展示总览，细节文档放专题页

## 九、结语

如果你也在做内部运维平台，建议把“权限、K8s、告警、日志”当作一条完整链路来设计，而不是拆成几个孤岛模块。  
`Yunshu` 这套实践的核心价值，就是**让排障路径更短、责任边界更清晰、上手成本更低**。

欢迎交流你们团队在平台化建设中的做法。
